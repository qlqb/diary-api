# 01. Product Plan

## 1. 앱의 한 줄 정의

diary-app은 사용자가 일상 기록 속에서 반복되는 패턴을 발견하고, 오늘 할 수 있는 작은 행동으로 조정해나가도록 돕는 자기관리 웹 애플리케이션이다.

이 앱은 사용자를 더 압박하는 생산성 앱이 아니다. 계획이 무너졌을 때 실패로 표시하고 끝내는 대신, 왜 밀렸는지 기록하고 다시 배치하거나 줄이거나 보류할 수 있게 돕는다.

## 2. 1차 사용자와 성공 기준

1차 사용자는 개발자 본인이다.

초기 성공 기준은 다음과 같다.

```text
개발자 본인이 30일 연속 매일 사용한다.
```

이 기준은 기능 개수보다 실제 생활 루프에서 계속 쓸 수 있는지를 검증하기 위한 기준이다.

30일 자가사용에서는 다음을 확인한다.

- 오늘 화면의 카드 구성이 실제로 충분한가
- 기록 입력 마찰이 낮은가
- viewMode가 실제 하루 운영에 맞는가
- 계획이 밀렸을 때 재배치/축소/보류 흐름이 자연스러운가

## 3. 핵심 방향

diary-app의 핵심 목적은 사용자의 생활 데이터를 한곳에 연결해, 반복되는 패턴과 다음 행동 후보를 발견하게 하는 것이다.

기록 대상은 일기, 감정, 수면, Todo, 일정 블록, 루틴, 지출처럼 여러 영역에 걸칠 수 있다. 다만 모든 기능을 각각 깊게 만드는 앱을 목표로 하지 않는다.

올인원 전략은 다음 의미로 유지한다.

```text
모든 기능을 전문 앱 수준으로 깊게 만드는 것이 아니라,
생활 데이터를 연결해 사용자의 패턴과 조정 지점을 보여주는 앱.
```

## 4. 핵심 차별점

### 4.1 실패 대신 재배치

계획이 완료되지 않았다는 사실만 저장하지 않는다.

사용자가 실제로 한 조정 행위를 이벤트로 남긴다.

- 완료
- 이동
- 축소
- 보류
- 삭제

이 이벤트 이력은 사용자를 평가하기 위한 기록이 아니라 다음 계획을 더 현실적으로 만들기 위한 데이터다.

### 4.2 수집은 가볍게, 분석은 깊게

초기 입력은 짧고 빠르게 한다.

- 오늘 컨디션
- 감정
- 수면
- 완료/이동/보류 같은 실행 결과
- 하루 마무리 메모

분석은 데이터가 쌓인 뒤에 깊게 한다. AI도 자동으로 개입하지 않고, 사용자가 명시적으로 요청했을 때 집계 데이터 중심으로 참고 후보를 제공한다.

### 4.3 상향식 목표 발견

처음부터 거대한 연간 목표를 세우는 방식보다, 실제 하루 기록에서 반복 행동과 패턴을 발견하는 방식을 우선한다.

기본 흐름은 다음과 같다.

```text
오늘 기록
→ 반복 행동
→ 주간 패턴
→ 월간 집중 후보
→ 장기 방향 후보
```

월간/연간/LifeGoal은 장기 확장 방향으로 남기되, MVP에서는 테이블과 화면을 만들지 않는다.

## 5. UX 원칙

- 기록은 사용자가 부담 없이 남길 수 있어야 한다.
- 분석은 사용자가 요청했을 때만 실행한다.
- AI 결과는 자동 저장하지 않고 후보로 보여준다.
- 사용자가 선택하거나 수정해 적용한 항목만 저장한다.
- 완료하지 못한 계획은 실패가 아니라 조정 대상으로 다룬다.
- 오늘 할 수 있는 작은 행동을 우선한다.

## 6. AI 원칙

AI는 판단자가 아니라 보조자다.

AI는 다음 역할을 할 수 있다.

- 기록 요약 후보 제공
- 주간 회고 요약 후보 제공
- 문제 후보 제공
- 행동 후보 제공
- 계획 후보 제공

단, AI 결과는 자동 저장하지 않는다. 사용자가 명시적으로 요청했을 때만 실행하고, 사용자가 확인한 뒤 적용한 항목만 저장한다.

일기 원문 전체를 기본 전송하지 않으며, 가능한 한 집계 데이터와 태그 중심으로 전달한다.
