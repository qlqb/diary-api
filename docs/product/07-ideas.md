# 07. 아이디어 주차장 (Ideas Parking Lot)

> 이 문서는 **결정이 아니라 기억**이다. 여기 적힌 것은 약속이 아니며, 갱신 의무도 없다.
> 규칙: append-only. 떠오르면 한 줄이라도 적는다. 정리는 주간 회고 때 한다.
> 기획서(01~06)로 승격되기 전까지 모든 미확정 아이디어는 여기 산다.
> 상태 표기: `검토 전` / `n차 이후 검토` / `보류` / `기각(사유)` / `승격(문서명)`

---

## 지출계획 AI 추천 (2026-07-05, 대화)

- 입력 방식은 **상담형이 아니라 기록형**. 근거: 사람은 자기 지출을 부정확하게 기억함(실제보다 30~40% 과소 보고). 상담형은 부정확한 기억이 입력이라 추천 퀄리티 상한이 낮음.
- 콜드 스타트는 온보딩 고정 질문 3~4개로 해결: 월 수입(대략) / 고정비 / 이번 달 큰 지출 예정. 1차 계획 골격은 AI 없이 규칙 기반으로 충분.
- 흐름: 온보딩(1회) → 지출 퀵로그 누적(15초 예산) → 월말 "지출계획 후보 받기"(사용자 트리거) → AI 후보 → 확인 → 수정 → 적용 → 실적이 다음 달 후보에 반영(15장 피드백 루프).
- 기존 설계에 새 개념 불필요: `ai_suggestions.suggestion_type = SPENDING_PLAN` 추가 수준.
- 토큰 비용 전략: **서버가 집계하고 AI는 해석만**. 원본 거래내역을 프롬프트에 넣지 않음(비싸고, 산수 틀리고, 개인정보 원칙 위반). 요청당 2~3천 토큰 수준 → 1인당 월 비용 원 단위. 안전장치 4종: 사용자 트리거 전용 / 결과 캐싱(ai_suggestions.content 저장, 같은 달 재요청 시 저장본) / 모델 티어링(집계 해석은 경량 모델) / 월 주기 기능이라 설계 자체가 rate limit.
- "토큰을 더 사서 감당"은 답이 아님. 비용은 설계 단계에서 해결. 비용이 문제 될 규모 = 수익화 시점.
- 상담형 채팅은 도입하더라도 "후보를 받은 뒤 조정하는 대화"로 축소. 장기.
- 상태: **3차(지출 기록) 이후 검토**

## 틈새 타깃 후보 (2026-07-05, 시장 분석 대화)

- 정면승부 금지: 가계부=토스/뱅크샐러드, AI 스케줄링=Motion/Reclaim, 올인원=히로/Notion, AI 일기=40+개. "생산성 앱 시장"이 아니라 틈새로 진입.
- 쐐기 후보 1: **한국 수험생/대학생 갓생 문화** — 열품타 계보. "무너져도 다시 잡는 앱" 메시지가 번아웃 담론과 부합.
- 쐐기 후보 2: **ADHD/실행기능 어려움 커뮤니티** — Motion조차 ADHD를 추천 대상으로 명시할 만큼 수요 검증됨. 이 앱의 설계(재배치 버튼, 자기비난 방지 언어, 초저마찰 입력, LIGHT 강도)가 우연히 이 집단에 정확히 부합.
- 공개 경로: 본인 30일 → 주변 5~10명(대학생) → 반응 시 틈새 커뮤니티 공개. 각 단계가 다음의 전제.
- 상태: **보류 (본인 30일 검증 통과 전에는 착수 금지)**

## 차별점 보호 원칙 (2026-07-05, 시장 분석 대화)

- 시장 경쟁 앱들의 공통 전제 = "더 생산적이 되고 싶다"(최적화). 이 앱의 전제 = "계획은 무너지고, 그건 실패가 아니다"(조정/회복). 같은 시장의 다른 앱이 아니라 다른 질문.
- 경쟁 앱이 안 하는 것 3개 = 이 앱의 존재 이유:
  1. **조정 이력의 1급 데이터화** (plan_item_events — 밀린 일이 빨간 뱃지가 아니라 분석 원천)
  2. **컨디션이 계획에 선행** (intensity + conditionTags)
  3. **도메인 교차** (수면 × 감정 × 이월 × 지출 — 한 그릇에 담긴 앱만 가능)
- 원칙: 구현 우선순위에서 이 3개는 절대 밀리지 않는다. 기능 추가 제안이 오면 "3개 중 무엇을 강화하는가"로 심사.
- 리스크 인지: 차별점이 전부 "써봐야 느껴지는" 종류라 스크린샷 마케팅이 어렵고 복사 가능. 해자는 기능이 아니라 축적된 개인 데이터와 루프.
- 상태: **원칙으로 상시 적용**

## 과거 계획 스냅샷 (2026-07-05, 이동 정책 논의)

- 진실의 원천 = `docs/product` 마크다운. docx는 마일스톤 스냅샷(현재 최신: v2.1, 2026-07-05).
- 캡처는 초저마찰(폰 메모/GitHub 이슈 한 줄), 정리는 주 1회 이 파일로. 이 앱의 철학과 동일: 기록은 가볍게, 정리는 나중에.
- AI 병행 운영: 발산(브레인스토밍, 초안, 설명)은 GPT / 수렴(설계 확정, 트레이드오프 판정, 구현, 문서 생성)은 Claude. GPT에 물을 때는 docs/product 핵심 + changelog를 컨텍스트로 제공하고 "확정 설계와 충돌하는 제안 금지" 명시.
- 대화에서 결정이 나오면 세션 끝에 changelog/ideas 형식으로 추출해 저장소에 커밋. docx 재발행은 v3급 마일스톤에서만.

## Pending 알림 정책 / 설정 (2026-07-08, 대화)

- pending은 이전 운영일의 미정리 ScheduleBlock이며, 실패가 아니라 정리 대상이다.
- 기본 방향은 다음 날 아침 pending 요약 알림이다.
- 사용자를 즉시 압박하지 않는 것이 기본값이다.
- 선택 설정으로 TIME_FIXED 블록의 end_time이 지난 뒤 즉시 알림을 제공할 수 있다.
- 즉시 알림은 TASK에는 적용하지 않는다. TASK는 start_time/end_time이 없기 때문이다.
- 설정 후보:
  - 알림 끄기
  - 다음 날 아침 요약
  - 시간이 지난 TIME_FIXED 즉시 알림
  - 둘 다
- 세부 설정 후보:
  - morningReminderTime
  - immediateReminderDelayMinutes
  - quietHours
- 실제 PWA notification / push notification 구현은 MVP 제외이며, 1차-D 이후 또는 별도 릴리스에서 검토한다.
- 상태: **1차-D 이후 검토**

## 완료/완료취소 반복 호출 장기 보안 전략 (2026-07-09, 보류)

- Redis 기반 rate limiting은 외부 사용자 공개 전 검토한다.
- Bucket4j 또는 Filter/Interceptor 기반 사용자별 요청 제한은 1차 MVP 이후 검토한다.
- AOP 커스텀 어노테이션 기반 초단기 중복 요청 차단은 필요성이 확인된 뒤 검토한다.
- 프론트 debounce/pending은 UX용이며 보안 대책이 아니다.
- 상태: **1차 MVP 이후 / 외부 사용자 공개 전 검토**
